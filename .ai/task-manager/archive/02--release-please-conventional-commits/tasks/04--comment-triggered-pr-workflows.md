# Create Comment-Triggered PR Workflows (Fast-Forward Merge and Rebase)

## Objective
Create two GitHub Actions workflows: one for true fast-forward merging via `/fast-forward` comment, and one for rebasing with empty-commit removal via `/rebase` comment.

## Skills Required
- `github-actions`: Authoring comment-triggered workflows with permission guards and PAT-based authentication

## Acceptance Criteria
- [ ] `.github/workflows/fast-forward.yml` exists and triggers on `/fast-forward` PR comment
- [ ] Fast-forward workflow uses `sequoia-pgp/fast-forward@v1` with `FF_MERGE_TOKEN`
- [ ] `.github/workflows/rebase.yml` exists and triggers on `/rebase` PR comment
- [ ] Rebase workflow uses `peter-evans/rebase` with `--no-keep-empty` option
- [ ] Rebase workflow has `author_association` permission guard (OWNER, MEMBER, COLLABORATOR)
- [ ] Both workflows use `FF_MERGE_TOKEN` repository secret

### .github/workflows/fast-forward.yml

```yaml
name: Fast Forward

on:
  issue_comment:
    types: [created]

jobs:
  fast-forward:
    if: >-
      github.event.issue.pull_request &&
      contains(github.event.comment.body, '/fast-forward')
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
    steps:
      - name: Fast-forward merge
        uses: sequoia-pgp/fast-forward@v1
        with:
          merge: true
          token: ${{ secrets.FF_MERGE_TOKEN }}
```

Key points:
- Triggers on `issue_comment` (not `pull_request_comment` â€” GitHub treats PR comments as issue comments).
- The `if:` condition checks both that the comment is on a PR (`github.event.issue.pull_request`) and that the body contains `/fast-forward`.
- `sequoia-pgp/fast-forward` enforces push-access internally â€” only users with push access can trigger the merge.
- Uses `FF_MERGE_TOKEN` (repo-level secret, NOT from the `release-please` environment) so that the push triggers downstream workflows like `release-please.yml`.
- If the branch cannot be fast-forwarded (not rebased on current `main`), the action posts a descriptive failure comment automatically.

### .github/workflows/rebase.yml

```yaml
name: Rebase

on:
  issue_comment:
    types: [created]

jobs:
  rebase:
    if: >-
      github.event.issue.pull_request &&
      contains(github.event.comment.body, '/rebase') &&
      (
        github.event.comment.author_association == 'OWNER' ||
        github.event.comment.author_association == 'MEMBER' ||
        github.event.comment.author_association == 'COLLABORATOR'
      )
    runs-on: ubuntu-latest
    steps:
      - name: Rebase PR branch
        uses: peter-evans/rebase@v3
        with:
          token: ${{ secrets.FF_MERGE_TOKEN }}
          rebase-options: '--no-keep-empty'
```

Key points:
- **Permission guard is critical**: Unlike `sequoia-pgp/fast-forward`, `peter-evans/rebase` has NO built-in permission check. Without the `author_association` guard in the `if:` condition, ANY GitHub user could comment `/rebase` on a PR and force-push the branch.
- `--no-keep-empty` drops commits that were empty from the start (e.g., auto-generated by GitHub Copilot), without affecting commits that have actual content.
- Reuses `FF_MERGE_TOKEN` â€” both workflows are bot-initiated pushes that need to trigger downstream CI and require `repo` scope.
- After the rebase force-push, the `conventional-commits.yml` workflow re-runs automatically on the updated branch.
- Typical workflow: `/rebase` to clean up empty commits â†’ wait for CI â†’ `/fast-forward` to merge.
